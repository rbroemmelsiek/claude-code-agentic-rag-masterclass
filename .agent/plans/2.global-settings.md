# Plan: Global Admin Settings + User Profiles

**Complexity:** ⚠️ **Medium** — well-defined pattern exists in `reference/CC-Rag-Tutorial-0.2/`; main project has no `user_settings` table to migrate. Touches DB, auth, API, and UI. Full removal of LLM/RAG env vars depends on Module 2 services (`llm_service`, `embedding_service`).

**Single-pass feasibility:** Phase A (profiles + admin + settings UI + Module 1 bridge) is realistic in one agent session. Phase B (env removal + embedding-dimension guard) should ship with Module 2 when `documents`/`chunks` exist.

---

## Clarification: where config lives today

| Storage | What | Plaintext? |
|---------|------|--------------|
| **`backend/.env`** | `OPENAI_API_KEY`, `OPENAI_VECTOR_STORE_ID`, Supabase keys, LangSmith | Yes — local file on the server, **not** in the database |
| **Database** | `threads`, `messages` only | N/A — no settings tables yet |
| **Per-user settings** | **Does not exist** in this repo | — |

The reference tutorial previously had a `user_settings` table; it was **replaced** by a single `global_settings` row plus `user_profiles.is_admin`. This plan follows that model.

**Why global, not per-user:** embedding vectors in `chunks` share one `embedding_dimensions` column. If users could pick different embedding models/dimensions, pgvector queries and stored chunks would be incompatible. LLM provider/model should also be consistent so retrieval + completion behavior is predictable.

---

## Target architecture

```text
┌─────────────────────────────────────────────────────────────────┐
│  Infrastructure secrets (stay in .env forever)                  │
│  SUPABASE_*, LANGSMITH_*, SETTINGS_ENCRYPTION_KEY, CORS_ORIGINS │
└─────────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────▼───────────────────────────────────┐
│  global_settings (single row, DB)                               │
│  LLM: model, base_url, api_key (encrypted)                      │
│  Embedding: model, base_url, api_key, dimensions (Module 2+)    │
│  RAG bridge (Module 1): vector_store_id, tool_choice            │
└─────────────────────────────────────────────────────────────────┘
                              │ read on every chat/ingest
┌─────────────────────────────▼───────────────────────────────────┐
│  All authenticated users use the same LLM + embedding config    │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  user_profiles (one row per auth.users)                         │
│  is_admin BOOLEAN — gates PUT /settings and Settings UI         │
│  Auto-created on signup via trigger (default is_admin = false)  │
└─────────────────────────────────────────────────────────────────┘
```

**Access model**

| Action | Non-admin user | Admin user |
|--------|----------------|------------|
| Chat, threads | ✅ | ✅ |
| `GET /settings` (masked keys) | ✅ | ✅ |
| `PUT /settings` | ❌ 403 | ✅ |
| Settings UI (`/settings`) | Hidden + redirect | ✅ |

## Phase A — Foundation (can build now, Module 1 still works)

### Task 1: Database migration

**File:** `supabase/migrations/20250607000000_global_settings.sql`

**Create `user_profiles`:**

```sql
user_id UUID PK → auth.users(id) ON DELETE CASCADE
is_admin BOOLEAN DEFAULT false
created_at TIMESTAMPTZ DEFAULT NOW()
```

- RLS: `SELECT` own profile only (`auth.uid() = user_id`)
- No user-facing `INSERT`/`UPDATE` policies — writes via `SECURITY DEFINER` trigger + service role

**Signup trigger:**

```sql
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE create_user_profile();
-- Inserts (user_id, is_admin=false) ON CONFLICT DO NOTHING
```

**Back-fill + admin bootstrap:**

- Insert profile for every existing `auth.users` row with `is_admin = false`
- Set `test@test.com` → `is_admin = true` (matches validation test users in `CLOUD.md`)
- Document how to promote additional admins: `UPDATE user_profiles SET is_admin = true WHERE user_id = ...`

**Create `global_settings` (single logical row):**

| Column | Purpose | Module |
|--------|---------|--------|
| `llm_model` | e.g. `gpt-4o`, `openai/gpt-4o` | 1+ |
| `llm_base_url` | OpenAI default; OpenRouter URL in M2 | 2+ |
| `llm_api_key` | Encrypted at rest | 1+ |
| `embedding_model` | e.g. `text-embedding-3-small` | 2+ |
| `embedding_base_url` | Provider URL | 2+ |
| `embedding_api_key` | Encrypted at rest | 2+ |
| `embedding_dimensions` | e.g. `1536` — **immutable once chunks exist** | 2+ |
| `openai_vector_store_id` | Module 1 managed RAG (`file_search`) | 1 (bridge) |
| `updated_at` | Audit | — |

- RLS: authenticated users can `SELECT`; no direct user `UPDATE`/`INSERT` (backend uses service role)
- Seed one empty row if none exists

**Bootstrap script (one-time, post-migration):**

**File:** `backend/scripts/seed_global_settings_from_env.py` (or admin-only API on first boot)

- Read current `OPENAI_API_KEY`, `OPENAI_VECTOR_STORE_ID` from `.env`
- Encrypt keys with `SETTINGS_ENCRYPTION_KEY`
- Write into `global_settings` if row is empty
- **Validation:** after seed, `GET /settings` returns masked keys; chat still works without `.env` LLM keys

**Validation test:**

## Security checklist

- [ ] API keys encrypted at rest (`SETTINGS_ENCRYPTION_KEY` in env, never in DB plaintext)
- [ ] Masked keys on `GET`; `PUT` ignores `***` masked values (don't overwrite with mask)
- [ ] `user_profiles` RLS: users read own row only; no self-promote to admin
- [ ] `global_settings` writes only via service role in backend
- [ ] Admin promotion only via migration, SQL, or future super-admin tooling — **not** via API self-service
- [ ] Rotate encryption key: document re-encrypt migration (future task)

## Migration path for existing deployments

1. Add `SETTINGS_ENCRYPTION_KEY` to `.env`
2. `supabase db push` — creates tables + trigger + back-fill
3. Run seed script to copy `OPENAI_API_KEY` / `OPENAI_VECTOR_STORE_ID` into `global_settings`
4. Deploy backend + frontend with settings UI
5. Admin verifies settings in UI; chat + RAG validation pass
6. Remove `OPENAI_API_KEY` and `OPENAI_VECTOR_STORE_ID` from `.env` (Phase B)
7. Restart backend

**Rollback:** keep env fallbacks in Phase A until step 6 is verified.
